# App Architecture

## Overview

The app uses SwiftUI + protocol-driven boundaries so core user flows can be tested without real Firebase or StoreKit services.

## Layer Map

```text
Views
  -> Flow ViewModels
    -> Managers
      -> Repositories / Services
        -> System APIs / Backend APIs
```

Cross-cutting:

- `Design/` for visual primitives
- `Utilities/Observability.swift` for logging/analytics abstractions
- `Config/AppConfig.swift` for runtime config

## Composition Root

`AppEnvironment` is the composition root.

Responsibilities:

- load `AppConfig`
- select logger/analytics implementations
- wire manager dependencies
- expose shared runtime objects to app entry points

## Current Production Wiring

- `AuthenticationView` uses `AuthenticationManager`
- `PaywallView` uses `SubscriptionManager`
- `AuthenticationManager` depends on `UserRepositorying`
- `UserRepository` depends on `UserStoring` (`DatabaseManager`)
- `SubscriptionManager` depends on `SubscriptionRepositorying`
- `StoreKitSubscriptionRepository` encapsulates StoreKit calls

## Testability Strategy

- UI flows are concentrated in:
  - `AuthenticationFlowViewModel`
  - `PaywallFlowViewModel`
- Managers and flows depend on protocols (`AuthenticationManaging`, `SubscriptionManaging`, `UserRepositorying`, `SubscriptionRepositorying`)
- Tests inject stubs/mocks and assert behavior without network or StoreKit setup
- Previews use `PreviewSupport/PreviewFixtures.swift` for deterministic states

## Data and State Patterns

- View models own UI state (`@Published`) and validation
- Managers own domain-level mutable state (`@Published`) and long-running workflows
- Repositories isolate data-source specifics
- Services isolate transport-level details (request/decode/errors)

## Configuration Pattern

`AppConfig` lookup order:

1. process environment
2. Info.plist values generated by xcconfigs
3. hardcoded fallback defaults

Keys:

- `APP_ENV`
- `API_URL`
- `GITHUB_TOKEN`

## Observability Pattern

Use abstractions, not direct SDK calls, in business logic:

- `Logging`
- `AnalyticsTracking`
- `AnalyticsEvent`

Default implementations:

- `AppLogger` (OSLog)
- `NoOpAnalyticsTracker`
- `ConsoleAnalyticsTracker` / `FirebaseAnalyticsTracker` via `AnalyticsTrackerFactory`

Replace `NoOpAnalyticsTracker` when adding product analytics.

## Accessibility and Motion

Design primitives and views respect accessibility settings where relevant:

- reduced motion handling for animated backgrounds/effects
- accessibility labels/hints for icon-only controls
- dynamic type support on key auth/paywall UI

## Boundary Rules

- Views should not call services directly
- Services should not contain business flow logic
- Models should remain data-centric
- Repositories/services should not own UI state
